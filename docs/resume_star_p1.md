# P1 · IDC 运维智能知识库助手 —— STAR 简历段

> 定位：面向 IDC 运维场景的 RAG 问答系统**原型**，用于验证 RAG 全栈方法论与工程判断力。
> 诚信原则：本段不写未实测/未发生的数字（无 hit@4 涨幅、无 P95 延迟、无 stars）——所有陈述均可经受面试追问。

---

## S · 背景

数据中心（IDC）运维知识高度分散在多份 PDF 手册（国标 GB 50174、机房白皮书、ITIL 摘要等）中，查阅耗时、口径不一、新人依赖老师傅口传，知识"沉在文档里、活在人脑里"、缺一个能精确溯源到条款的问答入口——这是行业公认的痛点。我在中国移动 IDC 有过行业实习背景，因此选取"运维知识库问答"作为切入点，自驱构建了这套 RAG 系统原型，作为面向 AI 解决方案工程师方向的实践项目。

## T · 任务

独立设计并实现一套面向 IDC 运维场景的 RAG 问答系统原型，覆盖数据接入、检索优化、生成质控、评估到部署上线的全链路，目标是让运维人员用自然语言拿到**带出处、可直接执行**的答案。

## A · 行动

1. **检索疑难根因定位**：针对高频问题"机房正压值"答案 chunk 始终无法进入 top-4，逐层拆解定位三环根因——多主题混切致向量语义稀释、排名沉至 52/871；BM25 混合仍失效因中文未分词，加入 jieba 分词后拉回 top-4；召回后仍拒答则因旧 prompt 规则过严。三环打通后稳定输出「正压值 10Pa（GB 50174 7.4.4 条）」。

2. **评估体系迭代升级**：初版评估集偏简单、hit@4 触顶导致 Reranker 等重排手段测不出增益——这一现象反过来促使我升级评估体系：为 25 道可答题补 chunk 级黄金标注（gold_span），并因 RAGAS 库与 LangChain 1.x 不兼容而自实现 context precision / recall、faithfulness、answer relevancy 四项指标，借此暴露 faithfulness=1.0 的统计盲区与约 30% 假性拒答。

3. **生成侧策略重构**：将"一刀切拒答"改为三层分级回答（可答即答／沾边给方向／确无覆盖才拒答），并在 prompt 植入引用溯源，实现「答案 + [来源: 文件名 / 条款号]」可溯源呈现。

4. **工程化交付**：以 FastAPI 流式接口、Docker Compose、loguru 全链路可观测中间件，将系统从"本地能跑的实验"沉淀为可上线、可部署、可监控的服务。

## R · 结果

- **攻克疑难 case**：从"检索不到 / 被拒答"到稳定输出带条款号的精确答案（如 10Pa，GB 50174 7.4.4）；
- **建立可信评估闭环**：自研四指标评估体系暴露出约 30% 假性拒答并据此修复，把"看起来的好指标"变成"经得起追问的真实水平"；
- **交付可上线服务**：FastAPI 流式 + Docker Compose + 全链路日志可观测；
- **沉淀方法论**：形成"指标好看先怀疑尺子、评估集是随系统迭代的活资产"的工程判断，已成为我做 RAG 的默认动作。

## 面试口头素材（不进简历正文，追问时用）

- **"库里多少文档？"** → "3 份公开国标和白皮书，这是个验证 RAG 全栈方法论的原型。加文档只是扩上游数据接入，不动核心的检索-评估-生成链路——这个项目要证明的是判断力，不是数据规模。"
- **查询改写取舍**：实现过 Multi-Query 与 HyDE，但 MQ 约 12s、HyDE 约 30s，双双击穿 IDC 场景"3 秒人工阅读线"红线，故砍掉不上线——**先定红线再拍取舍**。
- **reranker 那笔账**：BGE-reranker 成本约 1.5s（未破 3s 红线），但在当前评估集上测不出 hit@4 增益 → 结论"留，但先把评估集加难重测再定去留"——**不甩锅工具，先修尺子**。
- **评估钉子句**：「指标一好看，我第一反应是怀疑尺子——评估集难度合不合适？它是随系统迭代的资产，不是一次性交付物。」
