# P1 · IDC 运维智能知识库助手｜STAR 项目总结

## S · 场景

IDC 运维知识分散在国标、设备手册和行业白皮书中。新人处理问题时需要反复翻阅
PDF，检索速度慢，也难以快速确认答案对应的具体条款。我结合自己的 IDC 行业实习
经历，把运维知识查询作为 RAG 项目的落地场景。

## T · 目标

独立完成一套 RAG 服务原型，覆盖文档接入、切片、检索、生成、评测和 API 交付，
让用户能够通过自然语言获得带文件、页码和原文证据的回答。

## A · 实现

1. **定位检索与生成的复合问题**：针对“机房正压值”长期无法稳定回答的问题，
   分别检查切片排名、BM25 分词和生成 Prompt，确认多主题切片造成向量稀释、
   中文未分词影响 BM25，以及过度保守的拒答规则共同导致失败。
2. **建立分层评测**：构建 30 题 Golden Set，并实现 Context Precision、
   Context Recall、Faithfulness 和 Answer Relevancy。结合人工复查发现，
   Faithfulness 不能覆盖拒答质量，而 Answer Relevancy 暴露了约 30% 的假性拒答。
3. **完善证据契约**：为检索块生成稳定 `evidence_id`，约束回答只能引用本次上下文；
   API 返回文件名、页码、chunk 序号和原文预览。旧索引回填 ID 时不使用检索排名，
   避免同一证据跨查询发生漂移。
4. **完成服务化**：提供 FastAPI 普通/流式接口、Streamlit 页面、上传校验、
   向量库进程缓存、请求日志和 Docker Compose。

## R · 结果

- 在 30 题评测集上，`Hit@4` 从 96% 提升到 100%。
- 典型问题能够返回“10Pa（GB 50174 7.4.4 条）”及对应证据。
- 对 BM25、Reranker、Multi-Query 和 HyDE 完成对照实验；由于当前小语料未验证
  稳定收益，默认链路保留更简单的纯向量检索。
- 形成可部署、可追溯的 RAG 服务原型，并通过 3 项无模型测试验证证据契约。

## 项目边界

- 当前知识库规模较小，评测结论不直接外推到大规模生产环境。
- Docker 配置已经完成，但正式部署仍需补充鉴权、限流和密钥管理。
- 检索实验保留在 `experiments/`，是否进入主链路由评测结果决定。
