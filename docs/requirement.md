# IDC 运维智能知识库助手 · 需求设计文档

> Project 1 of《AI 解决方案工程师求职冲刺计划》
> 行业支点：中国移动 IDC 业务实习经验

---

## 1. 一句话定位

面向 IDC（互联网数据中心）一线运维人员的企业级 RAG 智能问答系统，把散落在数百份运维手册、SOP、国家标准中的知识，变成"一问即答、有据可查"的助手。

## 2. 用户画像

| 维度 | 描述 |
|---|---|
| 角色 | IDC 一线运维工程师，尤其是入职 1 年内的新人 |
| 场景 | 机房巡检、告警处置、夜班值守时需要快速查证操作规程 |
| 痛点 | 知识散落在 PDF 手册 / Wiki / 国标里，关键词搜不准、翻阅慢、问老员工有打扰成本 |
| 期望 | 用自然语言提问，秒级拿到带来源出处的可信答案；查不到时明确说"不知道"，而不是编 |

## 3. 场景故事（差异化叙事）

> ⚠️ 以下数字为占位符，请用你在中国移动 IDC 实习期间观察到的**真实情况**替换，
> 数字要经得起面试官追问（"这个 X 分钟是怎么得到的？"）。

在中国移动 IDC 业务实习期间观察到，一线运维新人遇到一个温控告警，平均要花 **【X】** 分钟才能在 **【200+】** 份分散的 PDF 手册和 Wiki 里找到对应的处置 SOP；遇到拿不准的，还得打电话求助带班老员工，夜班尤其低效。知识不是不存在，而是"检索成本"太高——这正是 RAG 能解决的典型场景：把非结构化文档变成可对话、可溯源的知识接口，让新人查 SOP 从 **【X】** 分钟降到秒级，同时减少对资深员工的打扰。

## 4. 核心功能（V1 范围）

- [x] 多格式语料接入（PDF 手册 / TXT SOP / 国标）
- [x] 文档切片 + 向量化 + 持久化（Chroma）
- [x] 语义检索 Top-K
- [x] 基于上下文的生成式回答 + 拒答机制
- [ ] （V2）混合检索 + Reranker + 查询改写（W2）
- [ ] （V2）FastAPI 流式接口 + Streamlit 前端 + Docker（W3）

## 5. 评估方式

- 30 题人工 golden set（事实/流程/判断/无答案四类）
- 指标：hit@4、人工 1–5 分、RAGAS 四项
- 所有版本数字沉淀到 `experiment_log.md`

## 6. 非目标（V1 不做，避免范围蔓延）

- 不做多租户 / 权限隔离
- 不做实时数据接入（监控、CMDB 留给 P2 工单 Agent）
- 不做模型微调（用 RAG + Prompt 解决）
