# 面向数据中心（IDC）行业的企业级 RAG 知识方案 · 需求设计文档

> Project 1 of《AI 解决方案工程师求职冲刺计划》
> 定位视角：以 AI 解决方案工程师身份，深入 IDC 这一垂直行业，把行业痛点翻译成可落地的 AI 方案
> 行业支点：中国移动 IDC 业务实习经验（B 端解决方案交付一线观察）

---

## 1. 一句话定位

一套面向**数据中心（IDC）行业客户**的企业级 RAG 知识方案：把客户散落在数百份手册、SOP、国家标准里的非结构化知识，转化为"一问即答、有据可查、查不到就拒答"的可信知识接口，显著降低客户的知识检索成本与新人培养成本。

> 作为方案设计者，我关注的不是"替某个岗位查资料"，而是"为这类 B 端客户交付一套能落地、能量化收益的 AI 知识系统"。

## 2. 客户与用户画像（解决方案视角）

| 维度 | 描述 |
|---|---|
| 买单方（客户） | 数据中心运营方 / IDC 业务线，关心交付效率、人力成本、知识资产沉淀 |
| 终端使用者 | 一线工程师，尤其入职 1 年内新人（高频查证操作规程的人群） |
| 业务痛点 | 知识散落在 PDF 手册 / Wiki / 国标里，关键词搜不准、翻阅慢、跨人求助有打扰成本，新人培养周期长 |
| 方案价值 | 自然语言提问→秒级拿到带来源出处的可信答案；查不到时明确拒答而非编造，把"检索成本"压到最低，可量化为人均查证耗时与培训成本的下降 |

## 3. 场景故事（差异化叙事 · 方案工程师口吻）

IDC 运维中一个行业公认的痛点：一线工程师遇到告警时，相关规程分散在多份手册、Wiki 和国标里，关键词搜不准、翻阅慢，拿不准时还得跨人求助，拉低整体效率、推高新人培养成本。知识不是不存在，而是"检索成本"太高。

这正是一个可被 AI 方案化的场景：把这套非结构化知识做成可对话、可溯源、可拒答的 RAG 知识接口——目标是让查证从"翻手册"变成"秒级拿到带出处的可信答案"，并以"带出处 + 不瞎编"建立对 AI 答案的信任。我有中国移动 IDC 行业实习背景、对该场景有基础认知，以此作为把 IDC 行业理解转化为 AI 落地能力的自驱实践项目。

## 4. 核心功能（V1 范围）

- [x] 多格式语料接入（PDF 手册 / TXT SOP / 国标）
- [x] 文档切片 + 向量化 + 持久化（Chroma）
- [x] 语义检索 Top-K
- [x] 基于上下文的生成式回答 + 拒答机制
- [ ] （V2）混合检索 + Reranker + 查询改写（W2）
- [ ] （V2）FastAPI 流式接口 + Streamlit 前端 + Docker（W3）

## 5. 评估方式

- 30 题人工 golden set（事实/流程/判断/无答案四类）
- 指标：hit@4、人工 1–5 分、RAGAS 四项
- 所有版本数字沉淀到 `experiment_log.md`

## 6. 非目标（V1 不做，避免范围蔓延）

- 不做多租户 / 权限隔离
- 不做实时数据接入（监控、CMDB 留给 P2 工单 Agent）
- 不做模型微调（用 RAG + Prompt 解决）
