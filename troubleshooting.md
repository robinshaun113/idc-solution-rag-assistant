# Troubleshooting · 卡点与解决记录

> 用法：卡壳超 1 小时、或踩到值得复盘的坑，就记一条。这是面试时讲"我怎么定位/解决问题"的弹药库。
> 每条按 现象 → 原因 → 解法 → 面试一句话 记录。

## 1. 百炼 embedding 报 400 `contents is neither str nor list of str`

- **现象**：调百炼 text-embedding-v2 报 400 InvalidParameter。
- **原因**：langchain-openai 默认把文本先用 tiktoken 切成 token 数组再发，但百炼 OpenAI 兼容接口只认字符串原文。
- **解法**：`OpenAIEmbeddings(check_embedding_ctx_length=False)`，关掉这个预处理。
- **面试一句话**：第三方 OpenAI 兼容接口和官方 SDK 有细微行为差异，要看实际请求体定位，而不是只看文档。

## 2. 改了 .env 的 Key 却不生效

- **现象**：修改 .env 里的 API Key，程序仍用旧值。
- **原因**：shell 里残留的同名环境变量优先级高于 .env 文件值，覆盖了文件。
- **解法**：`load_dotenv(..., override=True)`，强制用文件值覆盖环境变量。
- **面试一句话**：环境变量加载有优先级，配置不生效优先排查"是不是被更高优先级的源覆盖了"。

## 3. 真实 PDF 接入：扫描件 PyPDF 抽不出文字（2026/6/10）

- **现象**：接入 5 份真实 PDF 国标/白皮书后，`GBT51314数据中心基础设施运行维护标准.pdf`（18MB、43页）用 PyPDFLoader 抽取出 **0 字符**，其余 4 份正常（共约 19.4 万字符）。
- **原因**：该 PDF 是**扫描件（图片版）**，页面是图像而非文字层，PyPDF 这类文本抽取器抽不出内容。文件体积大（18M/43页 ≈ 0.4M/页）也是扫描件的典型信号。
- **解法（技术分层）**：先用 4 份文字版 PDF 跑通索引与评估；扫描件需要 OCR（光学字符识别）才能抽文字，归入 P3 多模态阶段用 PaddleOCR 统一处理，不在 P1 提前引入。
- **面试一句话**：接入真实文档时要先做"可解析性体检"——文字版直接抽、扫描件走 OCR，按文档类型做技术分层，而不是一个 loader 硬扛所有格式。

## 4. source 级 hit@4 造假——翻对了书但翻错了页（2026/6/10-11）

- **现象**：hit@4=96%~100%，但 5 道题答案明明在库里、generator 却拒答。表面看检索没问题。
- **原因**：hit@4 的判定逻辑只检查 `expected_source` 是否在 top-4 切片的 metadata 里——是 source 级（翻对书没），不是 chunk 级（翻对页没）。id5 的 GB50174 确实进了 top-4，但含"10Pa 静压差"的那个 chunk 排在第 52 名。
- **解法**：给评估集补 gold_span（每道题的答案原文片段），判定从"正确来源文件在不在 top-4"升级为"含答案的原文片段在不在 top-4"。
- **面试一句话**：评估指标的选择决定了你优化什么——hit@4(source 级)把自己骗到五轮实验才定位根因；必须用两级（source + chunk）交叉验证。

## 5. 假语料导致评估虚高 → 真语料接入后 hit@4 假性暴跌到 0%（2026/6/10）

- **现象**：接入真实国标语料、重建索引后，baseline 重跑 hit@4 从 100% 暴跌到 0%。
- **原因**：旧评估集的 expected_source 绑的是假语料文件名（假 SOP），真国标导入后检索返回正确文档但 expected_source 全是对不上的旧文件名。
- **解法**：重新基于真国标知识点手动重写 30 道 QA，expected_source 指向真实文档名。保留旧评估集作为备份。
- **面试一句话**：评估集必须与知识库同源——换了知识库必须同步校准评估集，否则指标骗人。

## 6. BM25 对中文完全失效——空格分词把整个 chunk 当做一个"词"（2026/6/15）

- **现象**：Day 9 纯 BM25 检索 hit@4=68%，远低于纯向量的 100%。以为是 BM25 不行，判了死刑。
- **原因**：`rank-bm25` 库默认按空格分词。中文没有空格，300 字的 chunk 被当成一个 token。BM25 的"词频-逆文档频率"机制完全报废——关键词"静压差"无法被识别为独立词。
- **解法**：引入 jieba 中文分词，传给 BM25Retriever 的 `preprocess_func`。分词后"静压差""10Pa""主机房"精准命中，id5 答案 chunk 从排名 52/871 跃至第 1。
- **面试一句话**：英文生态的开源工具在中文场景下必须先验证分词——"开箱即用"默认值是针对英文设计的。

## 7. Reranker 的盲区——粗排没进 top-20 的 chunk，精排救不了（2026/6/15）

- **现象**：Day 10 Reranker (BGE-reranker-base) 精排后 hit@4 仍 100%，延迟却从 167ms 涨到 1.5s，假性拒答题 id5/9 都没翻转。看似"Reranker 没用"。
- **原因**：Reranker 只能对粗排选出的候选做重排，不能从全库自选。id5 的答案 chunk 排 52 名，没进 top-20 → Reranker 根本看不见它。不是 Reranker 不准，是粗排没给它机会。
- **解法**：先修检索（BM25+分词把答案提进 top-4），再考虑是否加 Reranker。知道 Reranker 的能力边界——只在粗排候选池质量足够时才有用。
- **面试一句话**：Reranker 是精读不是全搜——粗排质量是它的天花板；先定位候选池问题再决定加不加精排。

## 8. HyDE 假答案带偏检索方向（2026/6/15）

- **现象**：Day 11 HyDE 的 hit@4 从 direct 100% 暴跌到 76%，判断题（id16-19）全军覆没。
- **原因**：HyDE 让 LLM 编一段假答案去检索。如果假答案偏向问题某一面而正确答案在另一面，假答案越自信偏得越远。判断题尤为典型——"是不是X"→ LLM 编了"是X"的证据，但正确答案在"不是X"那页。
- **解法**：当前小库不用 HyDE。大规模多领域库、用户问法和文档措辞差距大时可用——但必须验证假答案不偏。
- **面试一句话**：HyDE 的检索质量上界 = 假答案的准确性；假答案一偏、检索全偏。

## 9. RAGAS Faithfulness=1.0 是盲区——拒答不算编造但等于没帮助（2026/6/15）

- **现象**：自实现 RAGAS 四指标评估中 Faithfulness 恒为 1.0，但 Answer Relevancy 仅 0.71，约 30% 的题假性拒答（id5/8/9/10/13/16/20）。
- **原因**："该问题还未覆盖在知识库中"这句话本身忠于上下文（没有编造），F=1.0。但用户没得到答案。F 只测"有没有抄资料"，不测"答案有没有用"。
- **解法**：Faithfulness 必须配合 Answer Relevancy 一起看。AR 测"答案是否回应用户问题"——拒答题 AR=0。两个指标互补 → "不敢编但也不敢答"的真相浮出水面。
- **面试一句话**：单个评估指标有盲区——Faithfulness 满分不代表系统好（可能 30% 题在拒答），必须用多维指标交叉验证。

## 10. 生成层保守 prompt 导致假性拒答率 30%（2026/6/15）

- **现象**：五轮检索实验（切片/BM25/Reranker/查询改写/生成层）后，id5/9 仍拒答。hit@4=100%，上下文答案在库里，generator 就是不敢答。
- **原因**：旧 prompt 的拒答规则包含"部分沾边、不足以支撑准确回答的情况"→ 一刀切。LLM 看到上下文措辞和问题不完全一致（如"静压差"vs"气压差"、"正压"在答案里但问题没问）→ 判为"部分沾边"→ 拒答。
- **解法**：改为三层分级回答策略——a) 完整 → 直接答；b) 部分 → 先答能确定的，标注缺失；c) 无关 → 拒答。同时角色从"运维专家"升为"IDC 行业解决方案工程师"（匹配目标岗位叙事）。
- **面试一句话**：prompt 里的拒答阈值要分层设计——"部分沾边也拒答"在生产环境会把有用信息挡在门外，让用户觉得系统"什么都不敢说"。
