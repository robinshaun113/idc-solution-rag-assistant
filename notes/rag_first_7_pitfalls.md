# 我的第一个 RAG 系统：踩过的 7 个坑

> 草稿 · 2026/6/10 · 尹应善主笔
> 项目：idc-solution-rag-assistant（面向 IDC 行业客户的 AI 解决方案）
> 写法约定：每个坑按「现象 → 原因 → 解法 → 我学到的判断」四段。最后一段最值钱——它是把"踩坑"翻译成"判断力"的地方。

---

## 引子

这是我的第一个 RAG 系统：一个面向 IDC 行业客户的 AI 解决方案，把数据中心的国标规范（GB 50174、GB/T 2887）和厂商白皮书做成可问答、带出处、会拒答的知识助手。真正让我学到东西的不是"怎么把它搭起来"——那部分网上教程一大把；而是搭起来之后撞见的这些坑。代码可以查、可以抄，但"为什么这么判断、坏了怎么定位"是抄不来的。所以我把坑记下来，而不是记代码。

---

## 坑 1 · 百炼 embedding 报 400 `contents is neither str nor list of str`

- **现象**：调百炼 text-embedding-v2 报 400 InvalidParameter，请求被拒。
- **原因**：langchain-openai 默认会先用 tiktoken 把文本切成 token 数组、再发给接口（这是为适配 OpenAI 官方的行为）。但百炼的"OpenAI 兼容"接口只认字符串原文，收到 token 数组就报参数非法。
- **解法**：`OpenAIEmbeddings(check_embedding_ctx_length=False)`，关掉这步 tiktoken 预处理，让它直接发原文字符串。
- **我学到的判断**：第三方"OpenAI 兼容"接口 ≠ 官方行为，文档常对不上。定位这类问题要看**实际发出去的请求体**，而不是只读文档——差异往往藏在客户端默认做的预处理里。


## 坑 2 · 改了 .env 的 Key 却不生效

- **现象**：改了 .env 里的 API Key，重跑程序仍在用旧值。
- **原因**：shell 里残留了一个同名环境变量（之前 export 过），它的优先级高于 .env 文件值。`load_dotenv` 默认不覆盖已存在的环境变量，所以文件里的新值被旧的环境变量盖住了。
- **解法**：`load_dotenv(..., override=True)`，强制用文件值覆盖环境变量。
- **我学到的判断**：配置加载是有优先级的（环境变量 > 文件）。遇到"配置改了不生效"，第一反应不是怀疑文件没保存，而是排查"是不是被一个更高优先级的源覆盖了"。


## 坑 3 · 真实 PDF 是扫描件，PyPDF 抽出 0 字符

- **现象**：GBT51314 运维标准 PDF（18MB/43页）用 PyPDFLoader 抽出 0 字符，另 4 份文字版正常（约 19.4 万字符）。
- **原因**：这份 PDF 是扫描件——每一页是图像，没有文字层，PyPDF 这类文本抽取器只能读文字层，自然抽出 0 字符。文件体积大得反常（18M/43页 ≈ 0.4M/页）就是扫描件的典型信号。
- **解法**：先用 4 份文字版 PDF 跑通索引与评估；扫描件需要 OCR（光学字符识别）才能把图像转成文字，归入 P3 多模态阶段用 PaddleOCR 统一处理，不在 P1 提前引入。
- **我学到的判断**：接入真实文档前要先做"可解析性体检"——文字版直接抽、扫描件走 OCR，按文档类型做技术分层，而不是指望一个 loader 硬扛所有格式。先用能解析的跑通闭环、把难啃的分层延后，是控制项目节奏的判断。


## 坑 4 · 假语料的 baseline，指标虚假地好看

- **现象**：用自己编的假 SOP 做语料时，baseline hit@4 接近满分。
- **原因**：问题、语料、标准答案（expected_source）全是我一个人编的——自产自销。我照着某个知识点编假 SOP，又照着这个知识点出题，expected_source 自然指向我刚写的那份文件。检索器做的是一道"出题人和判卷人是同一个人"的题。hit@4 高，根本没证明系统检索能力强，只证明了"我编的题和我编的料是配套的"。库小让它更容易满分，但哪怕库很大，只要题和料同源自编，照样虚高。
- **解法**：把 4 份真实国标/白皮书 PDF 接进来替换假 SOP。关键不在"更多"，而在"不是我写的"——真国标的表述、术语、章节结构和我的提问不再天然对齐，检索这才第一次面对一个真实任务，指标才开始有意义。
- **我学到的判断**：指标可信的前提是"被测的真实数据独立于出题人"。语料、题、答案全自编时，我量的不是系统能力，是我自己的自洽——自洽当然接近 100%，但毫无意义。看到漂亮指标，第一反应该是警惕，先问它是在量能力还是在量自洽。


## 坑 5 · 评估集必须和知识库"同源"

- **现象**：换上真国标语料、重建索引后，旧评估集 hit@4 从满分暴跌到 0%。
- **原因**：旧评估集每道题的 expected_source 还硬编着假 SOP 的文件名，而那些假 SOP 文件已经不在库里了。于是检索器无论捞回哪 4 块（哪怕全是完美的真国标），那个旧文件名永远不可能出现在结果中 → 每道题机械必然地 miss → 必然 0%。这个 0% 既不在量系统、也不在量检索，它在量"那份已经不存在的旧文件出现了没"——答案永远是没有。系统没坏，是尺子坏了。
- **解法**：不是简单改个文件名，而是基于真国标的真实知识点整体重写 30 道题（旧集备份为 eval/qa_set_v1_fakeSOP.jsonl.bak）。因为光改 source 字段，题目内容还是冲着假 SOP 编的，对不上真国标的表述——评估集要整体跟着知识库重做。
- **我学到的判断**：从坑 4（假高）到坑 5（假低），我被指标骗了两次，方向相反：一次骗我说"很好"，一次骗我说"全坏了"。但 hit@4 这个数字的意义不是检索器给的，是 expected_source 这把"标准答案的尺子"给的——同一次检索，尺子不同，数字从 100% 翻到 0%。所以评估的本质是"拿一把尺子量系统"，前提是尺子本身得准：它必须独立（不是我自己捏造的）、真实存在（不是指向已删除的东西）、且和当前被测对象同步。"评估集与知识库必须同源"就是保证这把尺子可信的铁律。任何指标在你信它之前，先反问一句："这把尺子可信吗？"


## 坑 6 · 假性拒答——检索命中了，生成却拒答（hit@4 抓不到）

- **现象**：人工打分发现 id 5/8/9/16/20 五题 hit=1（检索命中、库里有答案），但生成层却回了"该问题还未覆盖在知识库中"。
- **原因**：hit=1 只保证"对的文件"进了 top-4，但一份国标切成上百块，命中的可能是封面、目录或邻近段落，真正含答案的那一小块没挤进前 4。LLM 看到的上下文里确实没有答案，于是触发拒答——这其实是拒答机制在**正常工作**（宁可不答也不瞎编），问题出在检索粒度，不在生成。它和"幻觉"正好相反：幻觉是没依据硬编，假性拒答是有依据却没捞到。
- **解法**：我没有当场修它，而是如实打了低分（2 分）留在 baseline 里。因为 baseline 是"优化前的诚实快照"，这些失败样本正是 W2 要打的靶子；若现在就改，基线被污染，后续优化就失去了干净的前后对比起点。
- **我学到的判断**：hit@4 只是第一道闸，它只对"该读的文件找到没"负责，不对"答案答出没"负责。站在用户视角，用户要的是答案，不是"检索器内部捞对了文件"这种他看不见的中间态——hit=1 却拒答，在用户眼里就是彻底失败，而 hit@4 报表会把它记成绿勾，**指标在骗你**。所以任何只报检索指标的 RAG 评估都是自欺，必须加人工/生成层这第二道闸，才能抓出"检索对了但答案没出来"的漏网之鱼。今天这第二道闸当场逮到 5 个。这是整篇最值钱的一坑。


## 坑 7 · 切片精度：chunk_size 不是随便设的（认知坑）

- **现象**：我一开始觉得切多大无所谓，能切开、塞得进模型就行。
- **原因/转变**：后来想通它是一个权衡——**切太大**，一个块塞进太多主题，块的平均语义被各种无关信息稀释，向量"谁都不太像"，检索精度掉；**切太小**，一条完整的规定被从中间切断（上半句在 A 块、下半句在 B 块），检索只捞回其中一块，LLM 拿到残缺上下文反而答不全。一个是"稀释"，一个是"断句"，最优值在两者之间。
- **解法/计划**：Day 8 网格扫 chunk_size∈{300,500,800,1200}×overlap∈{50,100,200} 共 12 组，用评估集量每组 hit@4。结果：最优 **chunk_size=300, overlap=50，hit@4 从 baseline 96% → 100%**。但更值钱的是一个反直觉发现：**hit@4 在多组都到了 100%，那 5 道假性拒答题却照样此起彼伏地拒答**——证明 hit@4 是 source 级（"对的书翻到没"）、不是 chunk 级（"对的页翻到没"），它根本量不出"答案小块进没进 top-4"。于是"检索命中却拒答"其实是两层：①答案块没挤进 top-4（检索侧，Day9/10 救）②块进来了但 prompt 太保守仍拒（生成侧，Day13 救）。详见 experiment_log.md exp01。
- **我学到的判断**：chunk_size 直接决定喂给 LLM 的小块语义准不准、和最终答案效果直接挂钩，而"多大算最优"切大切小各有代价、根本没法靠拍脑袋断定——只能用评估集去测。这正好接上坑 4-6：那三坑教我把"尺子"（评估集）校准，坑 7 就是拿这把校准好的尺子去定一个参数。评估不是一次性体检，是贯穿优化全过程的工具。

## 收尾

做这个项目之前，我以为做 RAG 难在搭管道——加载、切片、embedding、检索、生成，把这条链拼通。做完才发现，拼通不难，真正难的是**判断这套系统到底好不好**。

七个坑正好分三层：坑 1-3 是工程接入坑（接口差异、配置覆盖、扫描件），靠查证和细心就能填；坑 7 是调参坑，靠实验定。但中间的坑 4-6 才是这个项目真正教会我的东西——它们全和"评估"有关。我被指标骗了两次：自产自销的假语料让 hit@4 虚高到接近满分（坑 4），换真语料后旧评估集又让它机械地暴跌到 0%（坑 5）；而就算指标看着对，hit@4 命中了，生成层还可能假性拒答（坑 6）。

这三坑让我想通一件事：**指标不会自动可信，得先校准量它的尺子，而且一把尺子不够。** 评估集必须和知识库同源（尺子要准），检索之外还得有人工/生成层第二道闸（一把尺子量不全）。这套"不被指标骗、先问指标在量什么"的判断，比我写过的任何一行 RAG 代码都更难、也更值钱——因为代码会被 AI 商品化，判断不会。
