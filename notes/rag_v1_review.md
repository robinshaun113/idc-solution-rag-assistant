## 0. 一图看懂 RAG V1 全景

RAG = 先检索、再生成。让 LLM 回答前，先从知识库捞出相关资料塞给它，避免它瞎编。

```
【离线·建索引】 scripts/build_index.py 跑一次
   语料 → ①加载 → ②切片 → ③Embedding → ④存进 Chroma 向量库
         loaders   splitter  embeddings   vector_store

【在线·答问题】 main.py 每次提问
   问题 → ⑤检索相似切片 → ⑥拼上下文喂给 LLM 生成 → 带出处的答案
          retriever          generator
                    (rag_chain 把 ⑤⑥ 串起来)
```

## 一、检索侧

### 1. 为什么要切片（splitter）

**机制**：一个切片只讲一件事 → 它的向量含义「具体」，不被其他主题稀释 → 和对应问题的相似度更高、更易命中。反之整篇文档变成「一个大杂烩向量」，含义被一堆无关主题拉平，反而谁都不像。

次要好处：省 token、避免超出模型输入长度上限。但**主要理由是「提精度」**，别本末倒置。

> **面试版**：切片是为了让每个向量语义聚焦，不被无关主题稀释，从而提升检索命中精度；省 token 和避免超长是附带好处。

### 2. chunk_size 为什么要做调参实验（Day 8）

- 切太大 → 一个切片塞太多主题，含义被稀释，精度掉。
- 切太小 → 一句话被切断，上下文不完整，捞回来 LLM 也看不懂。
- 所以要网格扫 chunk_size 找平衡点，用前后对比数字说话。

> **面试版**：chunk_size 是「语义聚焦」和「上下文完整」的权衡，太大稀释、太小断句，所以要用评估集做实验找最优。

### 3. Embedding 怎么把「语义」变成「数字」

不是规则、不是关键词匹配，而是**训练出来的**。核心是**分布式假设**（Distributional Hypothesis）：*意思相近的词，出现的上下文也相近*。

训练链条（4 步）：
1. 拿海量文本（整个互联网）训练一个 embedding 模型。
2. 模型反复看到「温度过高」附近总跟着「散热/空调/告警」，「温控告警」附近也跟着「散热/空调/机房」。
3. 为了把这种「共同出现的规律」学进参数，模型被迫给这两段文本算出**方向相近的向量**（因为它们的「邻居」高度重叠）。
4. 于是「语义相近」被压缩成了「向量方向相近」，余弦相似度自然高。

> **面试版**：Embedding 是在海量语料上训练过的模型，基于分布式假设——经常出现在相似上下文里的文本，被映射到向量空间里方向相近的位置。所以字面不同、语义相近的两段话，向量也接近。

### 4. 余弦相似度 & 检索

检索 = 算「问题向量」和库里每个「切片向量」的余弦相似度，取最高的 k 个（V1 里 k=4）。
余弦相似度看的是**向量方向**夹角，方向越接近、相似度越高。

### 5. ANN：大规模检索为什么不暴力比对

- **暴力检索**：for 循环把问题向量和每个切片挨个算余弦。绝对准，但**几十万、上百万切片时慢到跪**。
- **ANN（Approximate Nearest Neighbor，近似最近邻）**：建索引时就把向量「组织好」（分堆 / 织成关系网），查询时只比一小部分，快速逼近答案。
  - 代价：偶尔不是绝对第一名，但够近（前几名）。
  - 为什么这代价能接受：RAG 只要最相关的几个喂给 LLM，第 4 和第 5 名差 0.001，谁进来 LLM 答得都一样。为那 0.001 多花 100 倍时间不值。
  - **进阶洞察**：向量越多越密集，第 1 名和第 N 名差距越小，「近似」代价越低 → **规模越大越该用 ANN、越不亏**（和暴力检索「规模越大越痛苦」完全反着来）。
  - Chroma 底层用的是 HNSW（一种基于图的 ANN）。**不用懂内部怎么建图**，记住「提前组织、查询少比」八个字即可。

> **面试版**：向量检索不是暴力两两比对，而是 ANN 近似最近邻（Chroma 底层是 HNSW），用极小的精度损失换大规模下的速度。

### 6. 向量数据库（Chroma）凭什么存在

如果只算余弦，numpy 几行就够，不需要向量库。Chroma 多干的是「大规模又快又稳地查」：
1. **ANN 索引**（核心、非它不可）：建好 HNSW 那套结构，把「比一百万次」变成「比几百次」。
2. 持久化：向量存磁盘（`./chroma_db/`），重启不用重算。
3. 增删改：新手册能单独加，过期的能删。
4. 元数据过滤：可带条件查（如只在某类别里检索）。

> **面试版**：小规模一次性检索 numpy 够了；要大规模、持久化、增删、又快又稳才需要向量数据库，它最核心的本事是用 ANN 索引解决暴力检索扛不住的规模问题。

### 7. 「学到什么程度算够」的判断（方案工程师定位）

ANN 内部怎么建图，是向量数据库/算法工程师的核心知识。对 AI 解决方案工程师 / FDE：
**懂它解决什么问题、付什么代价、什么时候用 —— 但不需要懂内部实现。**
被追问内部细节时，「我了解它是基于图的近似算法，建图细节没深入，但清楚它的取舍和适用场景」就是满分答案——展示了「知道能力边界在哪」。

## 二、生成侧

### 8. RAG 在对抗大模型的什么毛病：幻觉（Hallucination）

幻觉 = 大模型不懂装懂、一本正经地编。根源是它的本质是「猜下一个最像的词」，不是「查证事实」。

RAG 的两道防线：
- **第一道·喂资料**：回答前先检索，把真实资料塞进上下文 → 让它「开卷考试」，有据可依。
- **第二道·立规矩**：prompt 命令它「只准用给定资料，没有就拒答」→ 堵住「资料没有也硬答」的退路。

### 9. 拒答机制：不是功能缺陷，是「信任设计」

业务因果链（面试讲这个，不要只讲「幻觉不好」）：
> **坏答案 → 用户照做 → 业务损失/事故**。

更狠的视角：对 B 端客户，**一个自信的错误答案，比「我不知道」危险得多**。因为「不知道」用户会去核实，而「看起来对的错答案」用户会直接信——**信任一旦因一次错答崩掉，整个 AI 方案就被弃用**。所以拒答是「信任设计」。

本项目 prompt 里的拒答规则（生产级要点）：
1. 覆盖「半相关」：不只是「完全没有」，连「部分沾边、不足以支撑准确回答」也要归入拒答（否则 LLM 会用沾边资料硬凑）。
2. 给下一步：不要冷冰冰一句「不知道」，而是「该问题暂未覆盖在知识库中，建议咨询带班工程师」。
3. 禁止夹带：LLM 有「讨好」倾向，会先答一点再说找不到——规则要明确禁止这种混答。

> **面试版**：拒答不是缺陷而是信任设计，因为对 B 端客户一个自信的错答比「不知道」危险得多，会直接摧毁用户对方案的信任。

### 10. 为什么「拒答的判断」落在生成层，而不是检索层

关键：两层**掌握的信息不对等**。
- 检索层只会算相似度、返回「相对最像」的 k 个，给出的是**相对排名**，判断不了**绝对够不够**——因为「够不够回答」是个**语义理解**问题，它只会算数字距离。哪怕全库无关，它照样交上「相对最像」的 4 个。
- 生成层（LLM）**真读得懂**内容，能判断「这些片段跟问题没关系」→ 触发拒答。这需要语义判断力，正是 LLM 有、检索层没有的。

> **面试版**：检索层只能给「相对最像」、判断不了「绝对够不够」；「够不够回答」是语义理解问题，只有读得懂内容的生成层才能做，所以拒答天然落在生成层。

（实测：问「今天北京天气怎么样」，检索层照样召回了 4 个 SOP 片段，但生成层正确拒答——印证了分工。）

### 11. 双闸门拒答设计（Day 13 要补的升级）

只靠 LLM 自觉拒答不够稳（prompt 让它拒答它可能硬答）。更稳的是两道闸门：

| | 分数闸门（检索层） | 语义闸门（生成层） |
|---|---|---|
| 做法 | `if 最高相似度 < 阈值: 直接拒答`，不进 LLM | LLM 读内容判断够不够 |
| 性格 | 便宜、确定，但**笨**（只看分数读不懂内容） | 智能、读得懂，但**贵**（调 LLM、花 token）且**可能不听话** |
| 角色 | 在前面，零成本拦掉「明显无关」的大多数 | 在后面，兜住「分数过线但其实不对题」的漏网情况 |

为什么分两道：**便宜确定的在前拦大多数（省 LLM 调用），智能的在后兜漏网的边缘情况**。合成一道必然丢掉其中一个优点。本质是「成本 × 准确性」的权衡分层。

> **通用框架**：任何「多层防线」（限流、缓存、风控）都是「便宜的在前、智能的在后」。

## 三、今天的元收获（比知识点更值钱）

两个 AI 解决方案工程师的本能，要刻意保持：
1. **不盲从安排，先对齐目标**：动手做项目前先搞清楚「目标岗位到底要什么能力」。这是第 1 层「业务翻译力」的实战。
2. **想问题落到业务后果，而非模型指标**：讲拒答时讲「错答→照做→事故→信任崩塌」，而不是「faithfulness 低」。

目标岗位定位：AI 解决方案工程师需要同时具备业务沟通、方案设计和动手交付能力。
本项目用 IDC 行业场景展示从需求拆解到 RAG 原型、评测和服务化的完整过程。
